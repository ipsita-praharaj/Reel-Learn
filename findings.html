Planner failure analysis · h/web-surfer-flash baseline

The plan didn't fail from a bad idea. It failed from forty identical clicks.

A close read of one real, already-run session on H Company's baseline agent (holo3-1-35b-a3b) surfaces a specific, reproducible planner gap: no detection for interaction-level stuck loops, even though the same architecture handles environment-level blockers (login walls, bot checks) cleanly. Every claim below is a direct quote or count from the session's own trajectory — nothing inferred.

Primary evidence session d000f506-c039-4d31-9529-805ee6553552 Runtime 4m 24s · 76 steps · interrupted, no answer Cost 1,675,132 input tokens burned before external cutoff

The task: create a blank document in Photoshop on the Web and rename it to "Demo," using an agent whose instructions included a tutorial-derived step list ("click the document name, choose Rename, type the new name, press Enter"). The agent found the right menu. Then it never left it.

Finding 1confirmed, reproducible

No stuck-loop detection at the interaction level

Starting at step 53, the agent opens Document → Rename... in a hamburger submenu — visually exactly what its instructions describe. The click doesn't open a dialog. From there, across the next ~40 actions, the plan is identical every time: click near the same menu label, watch nothing happen, nudge the coordinates by a hair, repeat.

Step 57

"The click on 'Rename...' didn't seem to produce a visible dialog in the observation. Let me try clicking on the Rename option again, perhaps with slightly [different coordinates]."

That reasoning repeats, almost verbatim, for the rest of the session:

step 147 x=0.235 y=0.250 "Rename... text in Document submenu" step 151 x=0.260 y=0.255 "Rename... option text in Document submenu" step 179 x=0.245 y=0.255 "Rename... text in Document submenu" step 195 x=0.240 y=0.250 "Rename... text in Document submenu" step 231 x=0.260 y=0.255 "Rename... option text in Document submenu" step 263 x=0.235 y=0.250 "Rename... text in Document submenu" step 299 x=0.240 y=0.255 "Rename... text in Document submenu" step 307 x=0.245 y=0.255 "Rename... text in Document submenu" ← last action before external cutoff

Every one of ~20 consecutive attempts lands inside a box roughly 3% of screen width by 1.5% of screen height. The model narrates each as "let me try a different approach" — but by any reasonable definition, clicking the same label eight pixels to the left is not a different approach. It's the same approach, restated.

Concrete fix

A no-progress guard: if the last N actions (N≈4) target the same semantic element within a small coordinate tolerance and the resulting screenshot is unchanged, force either (a) a structurally different interaction — double-click, keyboard nav, scroll-then-click, hover-then-click — or (b) an honest "blocked" outcome. Cheap to implement, and it would have ended this session at step ~65 instead of 76+ (externally killed, never self-terminated).

Finding 2confirmed

Literal instructions anchor the model past its own better judgment

The stuck loop didn't start immediately — the agent actually tried several genuinely different tactics first: clicking the title directly (steps 29–41), the hamburger menu (45–53), F2 + retype (93–97), Ctrl+S (115, 159), and Save As (127–167). That's real exploration. The problem is what happened after: once "Document → Rename..." visually matched the literal wording of its instructions, the model stopped varying strategy and only varied pixels — even though that exact instructed action was the one demonstrably not working.

Step 53 — the anchoring moment

"The Document submenu is now open and I can see the 'Rename...' option at the top. This is exactly what I need to rename the document from 'Untitled-1'..."

A textual match against the instructions appears to outweigh the model's own accumulating disconfirming evidence (20+ failed clicks) that this path works. Earlier in the same session, less "correct-looking" attempts were abandoned after a single failure; this one wasn't abandoned after twenty.

Concrete fix

Weight repeated interaction failure above instruction-text similarity: a plan step matching visible UI text is a hint about where to look, not a guarantee the described action will register. Repeated failure of an instructed action should be stronger evidence than the instruction's own wording, and should trigger strategy-switching rather than reinforcing the current one.

Finding 3hypothesis — worth an engineering check

Possible coordinate-mapping issue on nested menu overlays

Across ~40 attempts spanning a 3%×1.5% region of the screen, the click on "Rename..." never once opened a dialog. If the element were genuinely at any of those coordinates, ordinary click noise should have landed correctly at least once. That it didn't raises a narrow, testable hypothesis: floating/nested submenu overlays in this specific web environment may have a coordinate transform offset between what the model sees (the downscaled screenshot) and what actually receives the click.

This one isn't proven by the trajectory alone — it's the kind of thing that needs a targeted repro on H's side, clicking known nested-menu coordinates and diffing against the DOM's actual hit-test.

Finding 4confirmed

Undetected loops are a token-cost bug, not just a UX bug

Every step resends the accumulating screenshot + reasoning history. This session alone burned 1,675,132 input tokens on the free-tier 35B model — almost entirely inside the stuck loop — before being cut off by the client's own timeout, not by anything on the agent's side. Finding 1's fix directly caps this: the earlier a loop is caught, the less it costs, and cost grows with every additional wasted retry.

Contrast

The same architecture handles a different failure type well

This isn't "the baseline can't detect failure" — it's narrower than that. Three other sessions in the same investigation show clean, fast, honest failure recognition, just for a different class of blocker: environment-level walls (login screens, bot checks) rather than interaction-level ones (a click that silently does nothing).

That's a much narrower, more tractable gap to close than "improve general robustness" — it's specifically about extending existing blocker-detection to interaction-level, not building failure detection from scratch.
